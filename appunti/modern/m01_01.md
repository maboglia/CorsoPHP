# Scheda pratica 01.01 (Corso PHP)

# Docker + DDEV/Lando + PHP-FPM/Nginx + CLI vs Web

Obiettivo della scheda: imparare a costruire un ambiente **identico alla produzione** (Linux + Nginx + PHP-FPM + MySQL) che si avvia con **un solo comando**, e capire **come cambia PHP tra Web e CLI**.

---

# 1) Docker & DDEV/Lando

## Concetto chiave

Docker permette di creare un **server virtuale leggero** (container) con:

* Linux
* Nginx
* PHP-FPM
* MySQL/MariaDB
* Redis (opzionale)
* strumenti (Composer, Node, ecc.)

Il vantaggio è che l’ambiente è **riproducibile**: stesso sistema su qualsiasi PC.

---

## Scenario pratico

Creiamo un progetto PHP in una cartella:

```
corso-php/
 ├── public/
 │    └── index.php
 ├── src/
 ├── composer.json
 └── .ddev/
```

---

## Esempio: Avvio con DDEV

### Step 1: installare DDEV (una volta sola sul PC)

(Dipende dal sistema operativo: Windows/Mac/Linux)

---

### Step 2: inizializzare il progetto

Dentro la cartella progetto:

```bash
ddev config
```

Risposte tipiche:

* project name: `corso-php`
* docroot: `public`
* php version: `8.2`
* webserver: `nginx-fpm`

---

### Step 3: avviare il server Linux identico alla produzione

```bash
ddev start
```

Risultato:

* viene creato un container Linux con Nginx + PHP-FPM + DB
* viene assegnato un URL tipo:

```
https://corso-php.ddev.site
```

---

### Step 4: test rapido

Crea `public/index.php`:

```php
<?php
phpinfo();
```

Poi apri il browser su URL DDEV.

---

## Comandi essenziali DDEV (da ricordare)

| Comando         | Scopo                      |
| --------------- | -------------------------- |
| `ddev start`    | accende il server          |
| `ddev stop`     | spegne i container         |
| `ddev restart`  | riavvia                    |
| `ddev describe` | mostra URL e porte         |
| `ddev ssh`      | entra nel container Linux  |
| `ddev logs`     | visualizza log             |
| `ddev delete`   | elimina progetto/container |

---

## Differenza tra Docker puro e DDEV/Lando

* **Docker puro**: devi scrivere tutto tu (docker-compose, immagini, reti…)
* **DDEV/Lando**: strumenti “pronti” per sviluppo web (molto più rapidi)

Per corsi e sviluppo rapido, DDEV è perfetto.

# 2) PHP-FPM & Nginx

## Concetto chiave

Con Apache tradizionale, PHP era spesso integrato come **modulo**.

Con Nginx invece PHP non gira dentro il web server: gira come **processo separato**.

### Architettura reale moderna

```
Browser → Nginx → PHP-FPM → PHP Script → risposta HTML/JSON
```

---

## Cos’è PHP-FPM

PHP-FPM significa:

**FastCGI Process Manager**

È un sistema che mantiene una “pool” di processi PHP pronti a eseguire script.

---

## Flusso pratico di richiesta

Quando scrivi:

```
https://sito.it/index.php
```

succede:

1. Nginx riceve la richiesta HTTP
2. capisce che il file è `.php`
3. inoltra la richiesta a PHP-FPM tramite FastCGI
4. PHP-FPM interpreta lo script e genera output
5. Nginx restituisce la risposta al browser

---

## Perché oggi si usa Nginx + PHP-FPM

* più veloce di Apache in gestione connessioni
* migliore scalabilità
* separazione dei ruoli (server vs interprete)
* gestione efficiente di molte richieste simultanee

---

## Esempio concettuale di configurazione Nginx (mini)

```nginx
location ~ \.php$ {
    fastcgi_pass php-fpm:9000;
    include fastcgi_params;
    fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
}
```

👉 Significa: “se arriva un file PHP, mandalo a PHP-FPM”.

# 3) CLI vs Web in PHP

## Concetto chiave

PHP può girare in due modalità:

* **Web (server mode)** → tramite Nginx/Apache
* **CLI (command line)** → da terminale

---

## Differenze fondamentali

| Aspetto       | Web                | CLI                |
| ------------- | ------------------ | ------------------ |
| input         | GET/POST/COOKIE    | argv / stdin       |
| output        | HTML/JSON          | testo su terminale |
| tempo massimo | limitato (timeout) | può durare ore     |
| contesto      | request HTTP       | processo locale    |
| uso tipico    | siti web / API     | script automazione |

---

## Test immediato CLI

Crea file `script.php`:

```php
<?php
echo "Ciao dal PHP CLI\n";
```

E lancialo:

```bash
php script.php
```

---

## Variabili disponibili

### In web:

```php
$_GET
$_POST
$_COOKIE
$_SESSION
$_SERVER
```

### In CLI:

```php
$argv   // parametri
$argc   // numero parametri
```

Esempio:

```php
<?php
echo "Parametri passati:\n";
print_r($argv);
```

Esecuzione:

```bash
php script.php mario 25
```

Output:

```
Array
(
  [0] => script.php
  [1] => mario
  [2] => 25
)
```

---

## Caso pratico reale: CLI per automazioni

Esempi tipici:

* backup database
* import CSV
* invio mail massive
* generazione PDF
* sincronizzazione dati
* cron job

# 4) Mini esercitazione completa (Web + CLI insieme)

## Struttura progetto

```
corso-php/
 ├── public/
 │    └── index.php
 ├── cli/
 │    └── test.php
 └── composer.json
```

---

### File WEB: `public/index.php`

```php
<?php
echo "<h1>PHP Web Mode</h1>";
echo "<p>Ora: " . date("Y-m-d H:i:s") . "</p>";
```

---

### File CLI: `cli/test.php`

```php
<?php
echo "PHP CLI Mode\n";
echo "Ora: " . date("Y-m-d H:i:s") . "\n";
echo "Argomenti:\n";
print_r($argv);
```

Esecuzione:

```bash
php cli/test.php prova 123
```

---

# 5) Collegamento diretto con Docker/DDEV

Con DDEV puoi eseguire PHP CLI *dentro il container*.

Esempio:

```bash
ddev ssh
php -v
php cli/test.php prova
```

Oppure senza entrare:

```bash
ddev exec php cli/test.php prova
```

👉 Questo è fondamentale perché significa che stai usando **lo stesso PHP della produzione**, non quello installato sul tuo PC.
