# 1) Constructor Property Promotion (PHP 8+)

## Problema storico (codice verboso)

Prima, per creare una classe semplice servivano molte righe:

```php
<?php
declare(strict_types=1);

class Studente {

    private int $id;
    private string $nome;
    private string $email;

    public function __construct(int $id, string $nome, string $email) {
        $this->id = $id;
        $this->nome = $nome;
        $this->email = $email;
    }

    public function getId(): int {
        return $this->id;
    }

    public function getNome(): string {
        return $this->nome;
    }

    public function getEmail(): string {
        return $this->email;
    }
}
```

---

## Soluzione moderna: Constructor Property Promotion

Ora puoi dichiarare le proprietà direttamente nel costruttore:

```php
<?php
declare(strict_types=1);

class Studente {

    public function __construct(
        private int $id,
        private string $nome,
        private string $email
    ) {}
}
```

👉 In una sola riga hai:

* proprietà dichiarate
* proprietà inizializzate
* costruttore completo

---

## Vantaggio pratico

Questo stile è perfetto per:

* DTO
* Entity semplici
* oggetti Value Object
* modelli per API REST

# 2) Readonly Properties (immutabilità controllata)

## Concetto

Una proprietà `readonly` può essere scritta **solo una volta**, tipicamente nel costruttore.

Dopo, non può più essere modificata.

---

## Esempio

```php
<?php
declare(strict_types=1);

class StudenteDTO {

    public function __construct(
        public readonly int $id,
        public readonly string $nome,
        public readonly string $email
    ) {}
}
```

Uso:

```php
$dto = new StudenteDTO(10, "Mario", "mario@test.it");

echo $dto->nome; // OK

$dto->nome = "Luigi"; // ERRORE!
```

👉 Questo garantisce che il DTO non venga alterato per sbaglio nel codice.

---

## Perché è fondamentale nei DTO?

Perché un DTO deve rappresentare un “pacchetto dati” affidabile:

* arriva dal DB → non deve cambiare casualmente
* arriva da una API → deve rimanere coerente
* passa tra Service e Controller → deve essere sicuro

# 3) Readonly Class (PHP 8.2+)

## Concetto

Una `readonly class` rende **tutte le proprietà readonly** automaticamente.

È la forma più forte di immutabilità.

---

## Esempio

```php
<?php
declare(strict_types=1);

readonly class EsameDTO {

    public function __construct(
        public int $id,
        public string $materia,
        public int $voto
    ) {}
}
```

Ora tutte le proprietà sono automaticamente `readonly`.

```php
$e = new EsameDTO(1, "Informatica", 28);

$e->voto = 30; // ERRORE
```

---

## Regola importante

Una readonly class può contenere solo proprietà tipizzate (sempre consigliato comunque).

---

# 4) Caso pratico completo: DTO “serio”

## Versione moderna (professionale)

```php
<?php
declare(strict_types=1);

readonly class UtenteDTO {

    public function __construct(
        public int $id,
        public string $username,
        public string $ruolo,
        public ?string $email
    ) {}
}
```

# 5) Mini esercitazione da laboratorio

## Obiettivo

Creare un DTO per gestire i dati di un prodotto.

### Specifiche

* id (int)
* nome (string)
* prezzo (float)
* categoria (string)
* descrizione opzionale (?string)

---

### Soluzione esempio

```php
<?php
declare(strict_types=1);

readonly class ProdottoDTO {

    public function __construct(
        public int $id,
        public string $nome,
        public float $prezzo,
        public string $categoria,
        public ?string $descrizione = null
    ) {}
}
```

Test:

```php
$p = new ProdottoDTO(1, "Mouse", 19.99, "Hardware");

echo $p->nome;

$p->prezzo = 10; // ERRORE: readonly
```

---

# 6) Quando usare cosa? (regola pratica)

### Constructor Property Promotion

📌 Usalo sempre per classi semplici: riduce codice inutile.

### Readonly Properties

📌 Usalo per DTO o oggetti che non devono cambiare dopo la creazione.

### Readonly Class

📌 Usalo per oggetti completamente immutabili (DTO “puri”).

---

# 7) Checklist finale (da ricordare)

✅ Constructor Promotion: meno righe, più chiarezza
✅ proprietà tipizzate sempre (`int`, `string`, `float`)
✅ DTO preferibilmente `readonly`
✅ evitare setter nei DTO
✅ usare `?string` per valori opzionali
✅ codice più sicuro e manutenibile
