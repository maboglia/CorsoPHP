## Scheda pratica (Corso PHP)

# 2.2 Astrazione e Interfacce: il “Contratto” (OOP serio in PHP)

Obiettivo: imparare a progettare codice **modulare e disaccoppiato** usando Interfacce, Classi Astratte e Traits.
Questa è la base per architetture moderne (MVC, Service Layer, Repository Pattern).

---

# 1) Concetto chiave: Astrazione = progettare bene

In PHP “professionale” non si programma pensando alle classi concrete, ma ai ruoli.

📌 Regola d’oro:

> “Programmare verso le interfacce, non verso le implementazioni.”

Questo significa: scrivere codice che dipende da un **contratto**, non da una classe specifica.

---

# 2) Interfaces (il contratto)

## Cosa sono?

Un’interfaccia definisce **cosa deve fare** una classe, ma non dice **come**.

Esempio tipico: un sistema di salvataggio dati.

---

## Esempio: Interfaccia Repository

```php
<?php
declare(strict_types=1);

interface StudenteRepositoryInterface {
    public function findById(int $id): ?StudenteDTO;
    public function save(StudenteDTO $studente): void;
}
```

Questa interfaccia dice:

* devi poter cercare uno studente per ID
* devi poterlo salvare

Non dice *dove* (DB, file, API).

---

## Implementazione concreta: MySQL

```php
class StudenteRepositoryMySQL implements StudenteRepositoryInterface {

    public function findById(int $id): ?StudenteDTO {
        // query su database
        return new StudenteDTO($id, "Mario", "mario@test.it");
    }

    public function save(StudenteDTO $studente): void {
        // insert/update su database
    }
}
```

---

## Implementazione alternativa: File JSON

```php
class StudenteRepositoryJson implements StudenteRepositoryInterface {

    public function findById(int $id): ?StudenteDTO {
        // lettura da file JSON
        return null;
    }

    public function save(StudenteDTO $studente): void {
        // scrittura su file JSON
    }
}
```

---

## Vantaggio reale (disaccoppiamento)

Il Service non deve sapere se i dati arrivano da MySQL o JSON.

```php
class StudenteService {

    public function __construct(
        private StudenteRepositoryInterface $repo
    ) {}

    public function getStudente(int $id): ?StudenteDTO {
        return $this->repo->findById($id);
    }
}
```

👉 Domani cambi DB?
Non tocchi il Service. Cambi solo l’implementazione.

# 3) Abstract Classes (blueprint con logica condivisa)

## Cosa sono?

Una classe astratta è una classe “incompleta” che:

* non può essere istanziata
* può contenere metodi già implementati
* può contenere metodi astratti obbligatori

Serve per evitare duplicazioni.

---

## Esempio: Repository astratto

```php
abstract class AbstractRepository {

    protected function log(string $msg): void {
        echo "[LOG] $msg\n";
    }

    abstract public function findById(int $id): mixed;
}
```

Ora una classe concreta eredita comportamento comune:

```php
class StudenteRepository extends AbstractRepository {

    public function findById(int $id): mixed {
        $this->log("Cerco studente con id=$id");
        return null;
    }
}
```

---

## Quando usare Abstract Class?

📌 Quando più classi condividono:

* metodi uguali
* proprietà comuni
* logica ripetuta (log, validazione, connessione DB, caching)

---

# 4) Traits (condivisione senza ereditarietà)

## Problema

PHP non supporta ereditarietà multipla:

❌ non puoi fare:

```php
class A extends B, C {}
```

Ma spesso vuoi “aggiungere funzionalità” a classi diverse.

---

## Soluzione: Trait

Un Trait è un blocco di metodi riutilizzabile.

```php
trait TimestampTrait {

    public function now(): string {
        return date("Y-m-d H:i:s");
    }
}
```

Uso:

```php
class OrdineService {
    use TimestampTrait;
}

class UtenteService {
    use TimestampTrait;
}
```

Entrambe le classi ottengono il metodo `now()`.

---

## Trait realistico: Logging

```php
trait LoggerTrait {

    public function logInfo(string $msg): void {
        echo "[INFO] $msg\n";
    }

    public function logError(string $msg): void {
        echo "[ERROR] $msg\n";
    }
}
```

Uso:

```php
class PagamentoService {
    use LoggerTrait;

    public function paga(float $importo): void {
        $this->logInfo("Pagamento avviato: $importo euro");
    }
}
```

---

# 5) Differenza importante: Interface vs Abstract vs Trait

| Strumento      | Serve a…              | Contiene codice?  | Obiettivo                        |
| -------------- | --------------------- | ----------------- | -------------------------------- |
| Interface      | definire un contratto | ❌ no (solo firme) | disaccoppiare                    |
| Abstract class | base comune + regole  | ✅ sì              | evitare duplicazione + struttura |
| Trait          | condividere metodi    | ✅ sì              | riuso senza ereditarietà         |

---

# 6) Composizione vs Ereditarietà (concetto chiave)

📌 OOP moderno preferisce la composizione.

### Ereditarietà (“è un”)

* `Studente extends Persona`
* forte dipendenza
* rigido

### Composizione (“ha un”)

* `Studente ha un Indirizzo`
* più flessibile
* più riusabile

👉 I Trait e le Interfacce aiutano proprio a evitare gerarchie rigide.

# 7) Mini esercitazione da laboratorio

## Obiettivo

Costruire un sistema di notifiche disaccoppiato.

### Step 1: definire l’interfaccia

```php
interface NotificaInterface {
    public function invia(string $destinatario, string $messaggio): void;
}
```

### Step 2: implementazione Email

```php
class EmailNotifica implements NotificaInterface {

    public function invia(string $destinatario, string $messaggio): void {
        echo "Invio EMAIL a $destinatario: $messaggio\n";
    }
}
```

### Step 3: implementazione SMS

```php
class SmsNotifica implements NotificaInterface {

    public function invia(string $destinatario, string $messaggio): void {
        echo "Invio SMS a $destinatario: $messaggio\n";
    }
}
```

### Step 4: service che usa l’interfaccia

```php
class NotificaService {

    public function __construct(
        private NotificaInterface $notifica
    ) {}

    public function avvisaUtente(string $user, string $msg): void {
        $this->notifica->invia($user, $msg);
    }
}
```

### Step 5: test

```php
$service = new NotificaService(new EmailNotifica());
$service->avvisaUtente("mario@test.it", "Benvenuto!");

$service2 = new NotificaService(new SmsNotifica());
$service2->avvisaUtente("3331234567", "Codice OTP: 9876");
```

👉 Il service non cambia. Cambia solo l’implementazione.

# 8) Checklist finale (da ricordare)

✅ usare **Interface** per disaccoppiare
✅ usare **Abstract Class** per condividere logica comune
✅ usare **Trait** per riuso di metodi trasversali
✅ preferire composizione all’ereditarietà
✅ scrivere servizi dipendenti da interfacce (Dependency Injection)
