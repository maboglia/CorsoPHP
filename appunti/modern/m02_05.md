## Scheda pratica (Corso PHP)

# 2.5 Introduzione alla Dependency Injection (DI)

Obiettivo: capire la Dependency Injection, applicarla in PHP e vedere perché è uno dei concetti più importanti per scrivere codice professionale, testabile e manutenibile.

---

# 1) Cos’è la Dependency Injection?

Una classe spesso ha bisogno di altre classi per funzionare.

Esempio: un `StudenteService` ha bisogno di un repository per leggere dati dal DB.

📌 Dependency Injection significa:

> Le dipendenze NON vengono create dentro la classe, ma vengono passate dall’esterno.

---

# 2) Esempio sbagliato (principiante): dipendenza creata internamente

```php
class StudenteService {

    private StudenteRepositoryPDO $repo;

    public function __construct() {
        $pdo = new PDO("mysql:host=localhost;dbname=scuola", "root", "root");
        $this->repo = new StudenteRepositoryPDO($pdo);
    }

    public function getStudente(int $id): ?StudenteDTO {
        return $this->repo->findById($id);
    }
}
```

## Problemi reali

❌ La classe è “incollata” a PDO e MySQL
❌ Impossibile cambiare repository senza modificare il codice
❌ Difficile da testare (richiede DB vero)
❌ Configurazione duplicata ovunque

---

# 3) Esempio corretto (professionale): Dependency Injection

```php id="ld0upf"
class StudenteService {

    public function __construct(
        private StudenteRepositoryInterface $repo
    ) {}

    public function getStudente(int $id): ?StudenteDTO {
        return $this->repo->findById($id);
    }
}
```

Ora il service non sa *come* funziona il repository.
Sa solo che esiste un contratto (`StudenteRepositoryInterface`).

---

# 4) Come si usa davvero? (Iniezione dall’esterno)

```php id="k8k5ja"
$pdo = new PDO("mysql:host=localhost;dbname=scuola", "root", "root");
$repo = new StudenteRepositoryPDO($pdo);

$service = new StudenteService($repo);

$studente = $service->getStudente(1);
```

👉 Questo è DI manuale (ma è già professionale).

---

# 5) Tipi di Dependency Injection

## 5.1 Constructor Injection (la migliore)

È quella vista sopra: la dipendenza è obbligatoria.

📌 È la più usata nei framework moderni.

---

## 5.2 Setter Injection (opzionale)

```php id="uwtpkv"
class StudenteService {

    private StudenteRepositoryInterface $repo;

    public function setRepo(StudenteRepositoryInterface $repo): void {
        $this->repo = $repo;
    }
}
```

⚠️ più rischiosa: puoi dimenticare di settare la dipendenza.

---

## 5.3 Method Injection (solo per un metodo)

```php id="nm0hkt"
class ReportService {

    public function generaReport(StudenteRepositoryInterface $repo): void {
        $lista = $repo->findAll();
    }
}
```

📌 utile quando la dipendenza serve solo in un caso specifico.

---

# 6) Vantaggio principale: TESTABILITÀ

Con DI puoi sostituire il repository reale con un repository finto (mock/stub).

---

## Esempio: Fake Repository per test

```php id="ywnu0y"
class StudenteRepositoryFake implements StudenteRepositoryInterface {

    public function findById(int $id): ?StudenteDTO {
        return new StudenteDTO($id, "TestUser", "test@test.it");
    }

    public function findAll(): array {
        return [];
    }

    public function save(StudenteDTO $dto): void {}
}
```

Ora testare il service è semplice:

```php id="z66jgm"
$repoFake = new StudenteRepositoryFake();
$service = new StudenteService($repoFake);

$studente = $service->getStudente(99);

echo $studente->nome; // TestUser
```

✅ Nessun DB
✅ Test velocissimo
✅ Test ripetibile

---

# 7) DI + Interfacce = Architettura pulita

La DI funziona bene solo se usi interfacce.

📌 Pattern tipico moderno:

* Controller dipende da Service
* Service dipende da RepositoryInterface
* RepositoryPDO implementa RepositoryInterface

Questo crea un sistema flessibile e scalabile.

# 8) Mini esercitazione da laboratorio (da corso)

## Obiettivo

Creare un sistema di login disaccoppiato.

---

### Step 1: Interfaccia repository utenti

```php id="h4km9l"
interface UtenteRepositoryInterface {
    public function findByUsername(string $username): ?UtenteDTO;
}
```

---

### Step 2: DTO

```php id="0jx0gv"
readonly class UtenteDTO {

    public function __construct(
        public int $id,
        public string $username,
        public string $passwordHash
    ) {}
}
```

---

### Step 3: Service con Dependency Injection

```php id="t6i5gj"
class AuthService {

    public function __construct(
        private UtenteRepositoryInterface $repo
    ) {}

    public function login(string $username, string $password): bool {

        $utente = $this->repo->findByUsername($username);

        if ($utente === null) {
            return false;
        }

        return password_verify($password, $utente->passwordHash);
    }
}
```

---

### Step 4: Fake Repository (test)

```php id="45a2s2"
class UtenteRepositoryFake implements UtenteRepositoryInterface {

    public function findByUsername(string $username): ?UtenteDTO {

        if ($username !== "mario") {
            return null;
        }

        return new UtenteDTO(
            1,
            "mario",
            password_hash("1234", PASSWORD_DEFAULT)
        );
    }
}
```

---

### Step 5: test finale

```php id="vutb5p"
$repo = new UtenteRepositoryFake();
$auth = new AuthService($repo);

var_dump($auth->login("mario", "1234")); // true
var_dump($auth->login("mario", "9999")); // false
var_dump($auth->login("anna", "1234"));  // false
```

---

# 9) Checklist finale (da ricordare)

✅ non creare dipendenze dentro la classe (`new` dentro service = sospetto)
✅ iniettare dipendenze tramite costruttore
✅ dipendere da interfacce, non da classi concrete
✅ testare con fake/mock repository
✅ codice più flessibile e scalabile
